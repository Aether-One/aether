# Onboarding — Aether

> This guide is for a human developer who has already run Aether locally and now needs to understand it well enough to change it safely. It explains the mental model and the *why*, not just the file layout.

## Why this project exists

Aether is a CLI that transforms any codebase into an "AI-native workspace" by analyzing a repository and generating a documentation knowledge base (architecture, modules, APIs, business rules, diagrams) using LLMs. The README states the motivation plainly: AI coding assistants are only as good as the context they receive, and most repositories have outdated docs, hidden business rules, and poor onboarding. Aether's stated mission is to build a self-explanatory, AI-ready knowledge layer from the repo itself.

The CONTEXT.md file frames Aether as a universe of lifecycle commands (`genesis`, `sync`, `doctor`, `explain`, `export`), where `genesis` is "the moment a project is born into the Aether ecosystem." That broader command set is **intent/roadmap**, not implemented reality — see the explicit notes below.

## Mental model

Aether is an interactive Node.js CLI (TypeScript, ESM). The core loop is:

1. `src/cli/index.ts` registers commands (`help`, builtins, config) and starts an interactive prompt via `startChat()` in `src/ui/prompt.ts`.
2. The user types a slash-command. `CommandRegistry` (`src/commands/registry.ts`) matches the command name case-insensitively and runs its `handler`.
3. `/genesis` (`src/commands/builtins.ts`) is the main feature: it loads AI provider config, pings the provider, scans the target project, asks the AI to *plan* which docs to generate, then generates each doc sequentially and writes them to `.aether/docs/`.
4. Provider access is unified behind the `LLMProvider` interface (`src/providers/types.ts`). The only implementation is `OpenAICompatibleProvider` (`src/providers/openai-compatible.ts`), created by `createProvider()` in `src/providers/factory.ts`. All providers (openai, gemini, anthropic, openrouter) are routed through that one OpenAI-compatible class.
5. Doc generation is driven by a fixed catalog in `src/genesis/docs.ts` (`DOC_DEFINITIONS`) plus AI-proposed custom docs. Each doc's prompt is assembled from `BASE_PROMPT` + project context + a specific prompt from `src/prompts/` + `PROMPT_SUFFIX`. The planner (`src/genesis/planner.ts`) decides which catalog docs apply and may propose up to 5 custom docs.

For the full file layout and architecture, see `architecture/system-overview.md` and `architecture/folder-structure.md` (generated by `/genesis`).

## Key decisions & the reasoning

- **Prompts live in `src/prompts/` as content, not inline code** (`CONTEXT.md` design decision). This lets prompts be iterated without touching logic. Each prompt file exports a constant (e.g. `SYSTEM_OVERVIEW_PROMPT`) consumed by `src/genesis/docs.ts`.
- **Planner runs before generation** (`src/genesis/planner.ts`). The AI decides which catalog docs fit the target project instead of generating all of them blindly (e.g. a frontend project should not get `api/endpoints.md` unless there is a real API/CLI).
- **Sequential, not parallel, doc generation** (`src/commands/builtins.ts` loop over `docsToGenerate`). Avoids overloading the provider API.
- **Provider-agnostic via OpenAI-compatible interface** (`src/providers/`). Swapping providers is a config change; `anthropic` is routed through `OpenAICompatibleProvider` with a `TODO` comment in `factory.ts` noting it needs its own provider due to a different API format.
- **No fallback between models** (`CONTEXT.md`, `src/commands/builtins.ts`): if the configured model fails, the error propagates and is formatted by `formatError()`.
- **Anti-hallucination "sandwich"** (`src/prompts/base.ts`): `BASE_PROMPT` is prepended and `PROMPT_SUFFIX` appended to every LLM call, instructing the model to only reference what is explicitly in the provided context and to treat vision docs (CONTEXT.md etc.) as intent, not proof of implementation.
- **`provider` field is re-synced from `baseUrl`** (`src/config/index.ts` `detectProviderFromBaseUrl`): when a user runs `/config set url <host>`, the `provider` label is corrected to match the known host, preventing silent drift. This was a explicitly "fixed recently" item in CONTEXT.md.
- **Do not change casually:** the `BASE_PROMPT` / `PROMPT_SUFFIX` rules and the `scanContext` budget logic in `src/genesis/context.ts` (no fixed file-count cap; `MAX_TOTAL_CHARS = 300_000`, `MAX_FILE_SIZE = 32_000`) — CONTEXT.md notes that an earlier fixed cap caused docs to be incomplete because files were silently dropped.

## Making your first change

A realistic small task: **add a new slash command**.

1. Open `src/commands/registry.ts` to see the `Command` interface (`name`, `description`, `usage?`, `handler`).
2. Create or edit a command file (e.g. add a registration call in `src/commands/builtins.ts` next to `registerBuiltinCommands`, or a new file imported by `src/cli/index.ts`). Use `registry.register({ name, description, usage, handler })`.
3. If the command needs AI access, follow the pattern in `builtins.ts`: `loadConfig(process.cwd())` → `createProvider(config)` → `chatWithRetry(provider, {...}, { onRetry: createRetryLogger() })`.
4. Register the command in `src/cli/index.ts` (e.g. `registerMyCommand()`) alongside `registerHelpCommand()`, `registerBuiltinCommands()`, `registerConfigCommand()`.
5. Run it: `npx tsx src/cli/index.ts` (the `dev` script in `package.json`), then type your command in the interactive prompt.
6. Verify: type `/help` — your command should appear with its `description` and `usage` (rendered by `src/commands/help.ts`). For type safety, run `npm run typecheck` (`tsc --noEmit`).

Another typical task: **tune a generated doc's prompt** — edit the relevant constant in `src/prompts/` (e.g. `SYSTEM_OVERVIEW_PROMPT` in `src/prompts/system-overview.ts`) and re-run `/genesis --force` on a test project.

## Gotchas

- **`/sync` is a stub.** `src/commands/builtins.ts` registers `sync` but its handler only prints "Still under development — not available yet." `/genesis` detects an existing `.aether/docs/` and points you to `/sync` while explicitly warning it is not available; use `/genesis --force` to regenerate. Do not assume incremental update works.
- **`anthropic` provider uses the OpenAI-compatible path.** `src/providers/factory.ts` returns `OpenAICompatibleProvider` for anthropic with a `TODO` comment; it is not a separate implementation.
- **Config is written to `.aether/config.json`** (via `saveConfig` in `src/config/index.ts`). CONTEXT.md notes this file was added to `.gitignore` to avoid committing API keys — but `.gitignore` is not present in the provided context, so verify locally before committing.
- **Commands are case-insensitive** (`registry.execute` lowercases the name), but doc output paths from the planner are sanitized in `src/genesis/planner.ts` (`sanitizeDocPath` rejects `..` and absolute paths). Don't trust raw LLM-proposed paths.
- **Local/small models hallucinate.** CONTEXT.md reports codestral and similar small local models ignore the anti-hallucination context; GPT-4o, Claude, and Gemini are the tested-reliable targets. This is a model behavior note, not a code guarantee.
- **No streaming output is shown to the terminal** during generation (`CONTEXT.md` design decision) — only the `StepRunner` spinner/writing/done states from `src/ui/steps.ts`.
